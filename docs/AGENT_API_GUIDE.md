# Vanyukov Furnace API — гайд для ИИ-агента

Документация для агента, который строит **интерфейс** и **симуляцию печи** поверх этого
API. Наша сторона — только API. Твоя сторона — UI + симулятор датчиков, отправляющий
данные в API.

## 1. Что это за система (контекст задачи)

Печь Ванюкова плавит медный концентрат. Боль оператора — **информационный лаг**: состав
шихты приходит из лаборатории через смену (ночную — к 10 утра), штейн/шлак — рентген с
задержкой. Онлайн есть только датчики. Система убирает лаг: по датчикам оценивает
шихту/штейн, ловит смену партии, детектирует отклонения и советует режим.

**Ключевой принцип входа:** почти все эндпоинты принимают `OnlineSnapshot` — срез из 22
датчиков, которые в реальности доступны в реальном времени. Лабораторные величины
(состав шихты, продукты) — это ВЫХОДЫ (что система оценивает), а не вход.

## 2. Запуск API

**Способ 1 — Docker (рекомендуется, ничего ставить не надо).** Модели `models/*.pkl`
уже внутри образа, обучение при старте НЕ идёт.
```bash
docker compose up            # поднимает API на http://localhost:8000
docker compose up --build    # если менял код/модели — пересобрать
docker compose down          # остановить
```

**Способ 2 — локально (нужен Python 3.11 + зависимости).**
```bash
uv pip install --python .venv/bin/python -r requirements.txt
PYTHONPATH=src .venv/bin/uvicorn api.main:app --port 8000
# модели уже в models/*.pkl; если их нет: PYTHONPATH=src python src/train.py
```

- Swagger UI (интерактивно): `http://localhost:8000/docs`
- OpenAPI JSON (для кодогенерации клиента): `http://localhost:8000/openapi.json`
- Проверка готовности: `GET /health` → `{"status":"ok", "models":[...]}`

> Все запросы к API одинаковы независимо от способа запуска — базовый URL
> `http://localhost:8000`. Твой UI/симулятор обращается к нему по HTTP.

## 3. Структуры данных

### OnlineSnapshot (главный вход, 22 поля, все float)

Рычаги оператора (дутьё и расходы):
`charge_flow`, `concentrate_flow`, `coal_flow`, `coal_dust_flow`, `addition_flow`,
`air_flow`, `oxygen_flow`, `oxygen_concentration`, `natural_gas_flow`

Состояние печи и котла:
`melt_temperature`, `melt_level`, `caisson_temperature`, `furnace_pressure`,
`offgas_temperature`, `offgas_flow`, `SO2_out`, `boiler_inlet_gas_temperature`,
`water_flow_circuits`, `feedwater_flow_boiler`, `steam_pressure_drum`,
`steam_temperature_drum`, `charge_moisture`

> Диапазоны каждого поля бери из `GET /ranges`. Реальные примеры — из `GET /sample`.

### ChargeComposition (опциональный вход `last_lab_charge`, 6 полей)

`Cu_charge`, `Zn_charge`, `CaO_charge`, `SiO2_charge`, `Fe_charge`, `S_charge`

Это последний известный лабораторный состав (прошлая смена). Передавать, если есть —
тогда оценка шихты и оптимизация используют его (точнее для непронаблюдаемых онлайн
компонентов). Наблюдаемость онлайн: **сера — сильно**, Fe — частично, Cu/Zn/CaO/SiO2 —
слабо (их и надо брать из last_lab_charge).

## 4. Эндпоинты — вход/выход

### Служебные (GET)

| Метод | Путь | Назначение | Выход |
|---|---|---|---|
| GET | `/health` | готовность | `{"status":"ok","models":[...]}` |
| GET | `/schema` | списки полей + наблюдаемость | см. ниже |
| GET | `/ranges` | min/median/max по входам | `{поле:{min,median,max}}` |
| GET | `/sample?n=5&tail=true` | реальные срезы из датасета | `{"samples":[OnlineSnapshot,...]}` |

`/schema` → `{"online_inputs":[...], "levers":[...], "online_state":[...],
"charge_composition":[...], "observable_online":{"strong":["S_charge"],
"partial":["Fe_charge"], "weak":[...]}}`.

---

### `POST /soft-sensor/matte` — Cu в штейне онлайн

Убирает лаг штейна (рентгенанализ). **Вход:** `OnlineSnapshot`. **Выход:**
```json
{"Cu_matte": 52.229}
```

---

### `POST /soft-sensor/charge` — оценка состава шихты

Убирает лаг шихты. **Вход:**
```json
{"online": {OnlineSnapshot...}, "last_lab_charge": {ChargeComposition...} | null}
```
**Выход:**
```json
{
  "charge": {"Cu_charge": 13.6, "Zn_charge": 4.4, "CaO_charge": 1.3,
             "SiO2_charge": 12.8, "Fe_charge": 27.6, "S_charge": 32.2},
  "source": "soft_sensor",          // или "lab_persistence" если дан last_lab_charge
  "soft_S_charge": 32.2,            // онлайн-оценка серы (независимый кросс-чек)
  "note": "серу видно онлайн; остальное точнее вести последним лаб-замером"
}
```

---

### `POST /twin/predict` — цифровой двойник

«Что будет, если…». **Вход:**
```json
{"levers": {9 рычагов...}, "charge": {ChargeComposition...}}
```
**Выход** (прогноз продуктов/состояния):
```json
{"Cu_slag": 0.85, "Cu_matte": 51.2, "melt_temperature": 1233.4, "SO2_out": 10.1,
 "melt_level": 1.516, "Fe_slag": 40.1, "SiO2_slag": 29.7, "Zn_slag": 4.9, "CaO_slag": 2.95}
```

---

### `POST /optimize` — рекомендация режима

Минимизирует `Cu_slag` + ресурсы при техограничениях, крутя рычаги в ±15% вокруг текущей
точки. **Вход:**
```json
{"online": {OnlineSnapshot...}, "last_lab_charge": {...} | null, "n_trials": 200}
```
**Выход:**
```json
{
  "recommended_levers": {9 рычагов — что выставить},
  "current_levers": {9 рычагов — как сейчас},
  "cu_slag_before": 0.929, "cu_slag_after": 0.856, "cu_slag_gain_pct": -7.9,
  "predicted_state_after": {Cu_slag, Cu_matte, melt_temperature, SO2_out, melt_level, ...},
  "feasible": true, "violations": []
}
```
> `n_trials` 100–200 — быстро (~1–2 c). Больше = точнее, медленнее.

---

### `POST /detect/anomaly` — отклонение режима

**Вход:** `OnlineSnapshot`. **Выход:**
```json
{"anomaly": false, "channel": "melt_temperature", "z": 1.23}
```
`z` — во сколько сигм факт отклонился от эталона; `anomaly=true` при z>5.

---

### `GET /demo/scenario?n=300&faults=3` — демо-сценарий с аномалиями

Наш датасет чистый (реальных сбоев нет), поэтому детектор аномалий на обычном
проигрывании молчит. Этот эндпоинт отдаёт реальные срезы с ЗАЛОЖЕННЫМИ отклонениями +
разметку — чтобы показать «раннее выявление отклонений» на демо. **Выход:**
```json
{
  "samples": [OnlineSnapshot, ...],          // проигрывать по порядку через /stream/step
  "injected": [
    {"start": 147, "end": 187, "channel": "melt_temperature", "type": "дрейф температуры расплава"},
    {"start": 170, "end": 200, "channel": "SO2_out", "type": "скачок SO2"}
  ],
  "note": "..."
}
```
Проигрывай `samples` через `/stream/step` — `regime_anomaly.anomaly` загорится на
участках `injected` (проверено: 3/3 участка, 0 ложных срабатываний). `injected` —
«правильные ответы» для подсветки в демо.

---

### `POST /stream/step` — полный проход (для симуляции)

Главный эндпоинт для симулятора датчиков: один срез → все оценки + детекторы + (опц.)
рекомендация. Детектор смены штабеля **stateful** — вызывать срезы последовательно.
**Вход:**
```json
{"online": {OnlineSnapshot...}, "last_lab_charge": {...} | null, "optimize": true}
```
**Выход:**
```json
{
  "matte": {"Cu_matte": 52.2},
  "charge": {"charge": {...}, "source": "...", "soft_S_charge": 32.2, "note": "..."},
  "regime_anomaly": {"anomaly": false, "channel": "melt_temperature", "z": 1.23},
  "charge_shift": {"charge_shift": false, "score": 0.4, "threshold": 1.8,
                   "estimated_composition": {"S_charge": 32.2, "Fe_charge": 27.6}},
  "recommendation": {как в /optimize}      // только если optimize=true
}
```

### `POST /stream/reset` — сброс буфера потока

Вызвать перед новой симуляцией. **Выход:** `{"status":"stream buffer reset"}`.

## 5. Как построить СИМУЛЯЦИЮ печи (датчики)

Наша задача — API; твоя — симулятор, отдающий срезы датчиков. Два способа:

**A. Проигрывание реальных данных (проще и достовернее).**
1. `GET /sample?n=500&tail=true` → 500 последовательных реальных срезов.
2. `POST /stream/reset`.
3. В цикле с интервалом (напр. 1 c = ускоренные 10 мин): бери следующий срез →
   `POST /stream/step` → показывай ответ в UI.
Это готовый «поток датчиков» без генерации.

**A+. Демо с аномалиями (чтобы показать детектор отклонений).**
На чистых данных детектор аномалий молчит. Для демонстрации используй
`GET /demo/scenario?n=300&faults=3` вместо `/sample` — те же реальные срезы, но с
заложенными сбоями. Проигрывай `samples` через `/stream/step`, подсвечивай красным при
`regime_anomaly.anomaly=true`. Массив `injected` — где именно заложены сбои (для сверки).

**B. Синтетический генератор датчиков.**
1. `GET /ranges` → min/median/max каждого поля.
2. Генерируй срез: значения вокруг median с шумом; держи автокорреляцию (следующее ≈
   предыдущее + маленький шаг), иначе детекторы будут шуметь.
3. Раз в ~48 шагов сдвигай «состав» — но состав не во входе (он оценивается), так что
   сдвигай коррелирующие датчики (`SO2_out`, `melt_temperature`) — тогда сработает
   детектор смены штабеля, наглядно для демо.
Способ A рекомендуется — меньше кода, данные физичны.

## 6. Как построить ИНТЕРФЕЙС оператора

Минимальный экран, обновляемый каждым `/stream/step`:

- **Датчики сейчас** — сырой `OnlineSnapshot` (что пришло).
- **Оценки без лаборатории** — `matte.Cu_matte`, `charge.charge.*` (подсветить серу —
  она онлайн), `charge.soft_S_charge`.
- **Алерты** — `regime_anomaly.anomaly` (красный + `channel`), `charge_shift.charge_shift`
  (баннер «смена штабеля — шихта поменялась»).
- **Рекомендация** — таблица `current_levers` → `recommended_levers`, крупно
  `cu_slag_gain_pct` («потери меди −X%»), бейдж `feasible`.

Рекомендация тяжелее детекторов (~1–2 c). Паттерн: `/stream/step` с `optimize:false`
каждый тик (быстрые оценки/алерты), а `/optimize` вызывать реже (кнопкой «советовать
режим» или раз в N тиков).

## 7. Частые ошибки

- **422 Unprocessable Entity** — не хватает поля в `OnlineSnapshot` или лишнее (`extra`
  запрещён). Шли ровно 22 поля из `/schema.online_inputs`.
- **Детектор смены не срабатывает** — не вызвал `/stream/reset` перед прогоном, или шлёшь
  срезы не последовательно. Детектор stateful, нужен непрерывный поток.
- **Оптимизатор не снижает Cu_slag** — на некоторых точках печь уже около оптимума,
  выигрыш мал. Это норма (не всякий режим можно улучшить).

## 8. Граница достоверности (важно)

Модели обучены на синтетическом датасете с заложенной физикой плавки
(`src/generate.py`). Метрики показывают, что метод восстанавливает эту физику — это
демонстрация, не валидация на реальной печи. Для продакшена нужна калибровка на
промышленных данных. В UI/презентации не подавать цифры как «доказано на производстве».

## 9. Что модели умеют и как это упаковать / продать

Раздел для того, кто делает презентацию и интерфейс: какую боль закрывает каждая фича,
что показать на экране, что говорить, какой цифрой подкрепить.

### Единый нарратив (elevator pitch)

> «Оператор печи Ванюкова работает вслепую: состав шихты знает из лаборатории через
> смену, штейн/шлак — рентген с задержкой. Между анализами крутит режим на глаз — отсюда
> потери меди и перерасход энергии. Наша система по онлайн-датчикам оценивает шихту и
> штейн, ловит смену партии и отклонения, и **сразу советует режим**. Один API —
> интерфейс и симуляция строятся поверх.»

Всё ниже — куски этой одной истории.

### Возможности → боль → как показать → что сказать → метрика

| Фича (эндпоинт) | Боль оператора | Что показать | Что сказать | Метрика |
|---|---|---|---|---|
| Soft-sensor штейна (`/soft-sensor/matte`) | Cu в штейне — только рентген с лагом | «Cu штейна сейчас: 52%» + редкая точка лаборатории | «штейн онлайн, не дожидаясь анализа» | точнее «последнего замера» на **83%** |
| Оценка шихты (`/soft-sensor/charge`) | состав шихты — через смену | «Сера: 32% (онлайн)», прочее — из прошлой смены | «серу видим сразу через SO2; остальное ведём и ловим смену партии» | сера R²=**0.78** |
| Детектор смены штабеля (`/stream/step`) | новую партию узнают к утру | баннер «⚠ смена штабеля» загорается сам | «пошла другая шихта — сигнал за 7 ч до лаборатории» | recall **100%**, лид **7.3 ч** |
| Детектор отклонений (`/demo/scenario`+`/stream/step`) | сбои замечают поздно | красный алерт «отклонение: melt_temperature», z растёт | «печь ведёт себя не как должна — видим в моменте, называем канал» | recall **83%**, 3/3 в демо |
| Двойник (`/twin/predict`) | нельзя экспериментировать на живой печи | меняешь рычаг → прогноз Cu_slag/темп/SO2 | «виртуальная печь: проверяем режим на модели» | rolling-CV R² 0.45–0.91 |
| Оптимизатор (`/optimize`) | режим на глаз → потери и перерасход | «сейчас → рекомендуем» + крупно «−X% потерь меди» | «конкретный совет: уставки в границах регламента» | `Cu_slag` **−51%**, ресурсы −6%, **100% в допуске** |

### Сценарий демо (последовательность экранов)

1. **Поток датчиков** — играешь `/stream/step` по `/sample`; живые оценки штейна/серы.
2. **Смена штабеля** — сам загорается баннер «другая шихта, лаборатория подтвердит через 7 ч».
3. **Переключаешь на `/demo/scenario`** — загорается отклонение режима с каналом.
4. **Кнопка «советовать режим»** (`/optimize`) — «потери меди −X%, вот новые уставки».
5. **Финал:** «всё — только по датчикам, без ожидания лаборатории. Один API, UI поверх».

### Ключевые цифры для слайдов

- **−51%** потери меди в шлак (на проблемных режимах), **100%** рекомендаций в техдопуске
- **+83%** точность оценки штейна между лабораторными замерами
- **7.3 ч** опережение лаборатории при смене партии сырья, recall **100%**
- **−6%** удельный расход ресурсов (кислород/газ/уголь)

### Как отвечать на неудобные вопросы (честность = сила)

- *«Это на реальных данных?»* → «На физически-корректном синтетическом датасете —
  демонстрация метода; архитектура готова, для продакшена нужна калибровка на промданных.»
- *«Восстанавливаете весь состав шихты?»* → «Онлайн честно видно серу (главный маркер);
  остальное ведём последним анализом — шихта кусочно-постоянна, а смену партии ловим
  детектором. Медь/флюсы физически не отражаются в датчиках — не выдумываем их.»
- *«Детектор аномалий на демо — сбои настоящие?»* → «Заложены нами намеренно: датасет
  чистый, показываем работоспособность детектора.»

Эта честность играет в плюс — показывает понимание физики, а не подгонку метрик.
