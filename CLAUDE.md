## Что это

Хакатон-проект: **прогнозирование и оптимизация режима плавки в печи Ванюкова**
(bath-smelting, переработка медного концентрата). Цель — рабочая, удобная модель для
демо; акцент на пользу для оператора, не на формальную метрику (её задаёт другой человек
при упаковке). «Переобучение не страшно» — но см. оговорку про Cu_slag ниже.

## Главный инсайт (со слов организаторов)

Основная боль — **информационный лаг**. Химсостав шихты оператор узнаёт из лаборатории
только через смену (ночную шихту — к 10 утра), штейн/шлак — тоже рентген/химанализ с
задержкой. Онлайн есть только датчики (дутьё, температуры, давление, уровень, газы,
расходы). Между анализами оператор работает «на глаз». Ценность системы — **убрать этот
лаг**: оценивать шихту и продукты онлайн по поведению печи и сразу советовать режим.

## Роли колонок

Разделение — по доступности в реальном времени (полный список в `src/config.py`):

- **Онлайн (датчики, сразу):** все `*_flow`, `oxygen_concentration`, `melt_temperature`,
  `melt_level`, `caisson_temperature`, `furnace_pressure`, `offgas_*`, `SO2_out`,
  `boiler_inlet_gas_temperature`, котёл (`*_drum`, `water_flow_circuits`,
  `feedwater_flow_boiler`), `charge_moisture` (допущение — уточнить).
  Из них **рычаги оператора** (крутит оптимизатор): расходы шихты/концентрата/угля/
  добавок, `air_flow`, `oxygen_flow`, `oxygen_concentration`, `natural_gas_flow`.
- **Лабораторное (с лагом, это таргеты):** состав шихты (`*_charge`) и продукты
  (`Cu_matte`, `Cu_slag`, `*_slag`). `Cu_slag` = потери меди в шлак — главный таргет.

Соглашение имён: `_charge` — в шихте, `_matte` — в штейне, `_slag` — в шлаке,
`_flow` — расход. `parameters.md` и диссертация (`mics/`) — источник истины по физике.

## Датасет — свой физический генератор (`src/generate.py`)

Исходная синтетика (`mics/dataset(in).csv`) физически неправдоподобна: состав шихты не
влиял на онлайн-сигналы, `Cu_slag` был нестационарен → 2 из 3 слоёв не работали. Мы
генерируем данные с каузально верной физикой плавки (те же 35 колонок), где состав шихты
влияет на продукты и наблюдаемые сигналы. `data/dataset.csv` — рабочая копия из генератора.
Оригинал сохранён в `mics/`. Метрики показывают, что пайплайн восстанавливает
заложенную физику (демонстрация метода). Регенерация: `python src/generate.py`.

Заложено: степень окисления `ox` ← кислород/расход; `Cu_matte` растёт с `ox`; `Cu_slag`
**U-образна** по `ox` (есть оптимум!); `SO2`/температура несут след состава (S, Cu →
состав наблюдаем онлайн); штабели (кусочно-постоянный состав) со сменами; связи
стационарны; инерция + шум.

## Что показали данные (подробно — `reports/findings.md`)

- `Cu_slag` **стационарен**: time-split R²=0.74 (на старых данных был −0.29).
- Состав шихты **наблюдаем онлайн**: soft-sensor `S_charge` R²=0.78 (через SO2).
- `Cu_slag` **предсказуем и управляем** рычагами (детренд-R²=0.41) → оптимизация имеет смысл.
- Состав шихты **кусочно-постоянен** («штабель») → смену партии детектируем по
  change-point на оценке состава soft-sensor'ом (опережение лаборатории ~7 ч).

## Целевая архитектура (3 слоя)

1. **Оценка шихты:** персистентность последнего лаб-значения + детектор смены штабеля.
2. **Soft-sensor продуктов:** `Cu_matte` — глобальная модель; шлак — rolling + калибровка.
3. **Цифровой двойник + Optuna:** двойник `f(рычаги, оценённый состав) → продукты/ресурсы`
   как локальный интерполятор; Optuna крутит рычаги в **bounds по train-диапазонам** →
   минимум `Cu_slag` + расход ресурсов при ограничениях (`melt_temperature`, `Cu_matte`,
   `SO2_out`, `melt_level` — см. `CONSTRAINTS` в `config.py`).

## Валидация

- **Только хронологический сплит / rolling-origin** — никакого случайного перемешивания
  во времени (утечка). Random-CV используем лишь как диагностику «выучиваемости».
- Оптимизатор держать в границах наблюдённых данных, иначе двойник экстраполирует в мусор.

## Окружение и запуск

Python 3.11 в `.venv` (uv). LightGBM требует `libomp` (`brew install libomp`).

```bash
uv venv --python 3.11 .venv                      # создать окружение
uv pip install --python .venv/bin/python \
  pandas numpy scikit-learn lightgbm optuna matplotlib seaborn

# запуск скриптов — из корня проекта, модули лежат в src/
PYTHONPATH=src .venv/bin/python src/generate.py       # сгенерировать датасет (физика)
PYTHONPATH=src .venv/bin/python src/eda.py            # EDA + решаемость обратной задачи
PYTHONPATH=src .venv/bin/python src/diagnostics2.py   # стационарность, вклад лагов
PYTHONPATH=src .venv/bin/python src/controllability.py # управляемость выходов рычагами
PYTHONPATH=src .venv/bin/python src/twin.py           # обучить двойник + rolling-CV
PYTHONPATH=src .venv/bin/python src/optimize.py       # оптимизация режима (Optuna)
PYTHONPATH=src .venv/bin/python src/soft_sensor.py    # soft-sensor штейна (Cu_matte)
PYTHONPATH=src .venv/bin/python src/anomaly.py        # детектор аномалий + смены штабеля
PYTHONPATH=src .venv/bin/python src/pipeline.py       # СКВОЗНОЙ поток (end-to-end)
```

## Сквозной поток (`src/pipeline.py`) — единая точка входа

`FurnacePipeline` связывает все слои в цепочку, работающую БЕЗ лабораторных данных на
входе: `online-датчики → оценка шихты → оценка штейна → двойник → рекомендация + флаги`.
Метод `run(snapshot, last_lab_charge)` принимает онлайн-срез и возвращает всё сразу
(оценки, рекомендованные рычаги, прогноз Cu_slag, допустимость, флаги). Модель —
`models/pipeline.joblib`, демо-результат — `reports/pipeline_result.json`.

**Как решается шихта (честно):** состав кусочно-постоянен, поэтому основная оценка —
персистентность последнего лаб-замера (точнее онлайн soft-sensor'а для всех компонентов,
кроме серы). Сера наблюдаема онлайн (R²=0.78) — независимый кросс-чек. Детектор смены
штабеля флагует новую партию за ~7 ч до лаборатории. Оптимизатор питается этим
оценённым составом, а не истинным — это и есть работа «как на реальной печи».

## Статус (три рабочих слоя, метрики — `reports/findings.md`)

1. **Soft-sensor штейна** (`src/soft_sensor.py`) — `Cu_matte` онлайн по датчикам без
   лаборатории. Между замерами точнее персистентности на **83%**, R²_last 0.95.
2. **Цифровой двойник + оптимизатор** (`src/twin.py`, `src/optimize.py`) — Optuna крутит
   рычаги в локальной окрестности (±15% ∩ перцентили), минимизирует `Cu_slag` + ресурсы
   при ограничениях. На проблемных режимах: `Cu_slag` −**45%**, ресурсы −13%, 100% в допуске.
3. **Детектор** (`src/anomaly.py`) — смена штабеля (через оценку состава soft-sensor'ом):
   recall **100%**, опережение лаборатории **7.3 ч**; отклонения режима (эталон vs факт,
   инъекционная валидация): recall 83%.

Артефакты: `models/*.joblib`, `reports/*_result.json`, `reports/figures/*.png`.

## Структура (ML-проект + API)

```
data/dataset.csv     # основной датасет (из src/generate.py); оригинал — mics/
src/                 # ядро: config, data, generate, soft_sensor, twin, optimize,
                     #       anomaly, detectors (онлайн), pipeline (end-to-end), train
models/*.pkl         # артефакты: pipeline.pkl, anomaly_detector.pkl, shift_detector.pkl
api/                 # FastAPI: main.py (эндпоинты), schemas.py, service.py
docs/AGENT_API_GUIDE.md  # гайд для ИИ-агента: вход/выход API, UI, симуляция датчиков
reports/             # findings.md, figures, *_result.json
deprecated/          # исследовательские скрипты (eda, diagnostics*, controllability)
mics/                # исходники: dataset(in).csv, parameters.md, диссертация (PDF)
```

Модели обучаются `src/train.py` → `models/*.pkl`. API грузит их и обслуживает слои;
детектор смены штабеля stateful (`src/detectors.py`) для потоковой симуляции.
Запуск API: `PYTHONPATH=src .venv/bin/uvicorn api.main:app --port 8000` (Swagger `/docs`).
Целевая функция оптимизатора нормируется на масштаб (`optimize.objective_value`), иначе
крупные ресурсы подавляют `Cu_slag` — веса в `config.OBJECTIVE_MINIMIZE` это приоритеты.
