# 🔥 Печь Ванюкова — прогноз и оптимизация режима плавки

Система убирает **информационный лаг** оператора медеплавильной печи: по онлайн-датчикам
оценивает состав шихты и штейн (не дожидаясь лаборатории), ловит смену партии сырья и
отклонения режима, и советует уставки для снижения потерь меди в шлак и расхода ресурсов.

> **Стек:** Python 3.11 · LightGBM · Optuna · FastAPI · Docker
> **Всё в API** — интерфейс и симуляция строятся поверх (см. [гайд](docs/AGENT_API_GUIDE.md)).

---

## Проблема

Оператор печи Ванюкова работает почти вслепую:

- **Состав шихты** приходит из лаборатории через смену — ночную партию узнают к 10 утра.
- **Штейн и шлак** — рентгенанализ с задержкой.
- Онлайн доступны только датчики (дутьё, температуры, давление, газы, расходы).

Между анализами режим крутят «на глаз». Отсюда — нестабильное качество, потери меди со
шлаком и перерасход кислорода/топлива.

## Решение

Шесть слоёв, собранных в один сквозной поток `онлайн-датчики → оценки → детекторы → рекомендация`:

| # | Слой | Что делает | Метрика |
|---|------|-----------|---------|
| 1 | **Soft-sensor штейна** | `Cu_matte` онлайн без лаборатории | точнее «последнего замера» на **83%** |
| 2 | **Оценка шихты** | сера онлайн + персистентность состава | сера R²=**0.78** |
| 3 | **Детектор смены штабеля** | ловит новую партию сырья | recall **100%**, за **7.3 ч** до лаборатории |
| 4 | **Детектор отклонений режима** | факт vs эталон, указывает канал | recall **83%** |
| 5 | **Цифровой двойник** | `f(рычаги, состав) → продукты` | rolling-CV R² 0.45–0.91 |
| 6 | **Оптимизатор режима** | Optuna: минимум потерь + ресурсов | `Cu_slag` **−51%**, ресурсы −6%, **100% в допуске** |

## API

Все входы — онлайн-срез датчиков (`OnlineSnapshot`, 22 поля). Swagger: `/docs`.

| Метод | Эндпоинт | Назначение |
|-------|----------|-----------|
| POST | `/soft-sensor/matte` | оценка Cu в штейне |
| POST | `/soft-sensor/charge` | оценка состава шихты |
| POST | `/twin/predict` | прогноз двойника |
| POST | `/optimize` | рекомендация режима |
| POST | `/detect/anomaly` | отклонение режима |
| POST | `/stream/step` `/stream/reset` | потоковый режим (для симуляции) |
| GET | `/demo/scenario` | срезы с заложенными сбоями (демо детектора) |
| GET | `/schema` `/ranges` `/sample` `/health` | служебные |

## Как это работает

Датасет сгенерирован с **каузально верной физикой** плавки (`src/generate.py`): состав
шихты влияет на продукты и наблюдаемые сигналы (сера → SO2, окисление → штейн,
U-образная зависимость потерь меди от степени окисления → есть оптимум). Модели —
LightGBM на честном хронологическом сплите; оптимизатор крутит рычаги в локальной
окрестности текущей точки, не экстраполируя в нефизичную зону. Подробности —
[reports/findings.md](reports/findings.md).

## ⚠️ Граница достоверности

Модели обучены на синтетическом датасете с заложенной физикой — метрики показывают, что
**метод восстанавливает эту физику** (демонстрация подхода), а не валидацию на реальной
печи. Для продакшена нужна калибровка на промышленных данных. Из состава шихты онлайн
физически наблюдаема **сера**; остальное ведётся последним лабораторным замером
(состав кусочно-постоянен) + детектор смены партии.
